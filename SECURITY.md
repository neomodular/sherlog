# Security Policy

## Sherlog's security model

Sherlog is a local development tool. Its trust boundary is your machine:

- The daemon listens on `127.0.0.1:2218` only — never on external interfaces.
- Debug session paths use unguessable random IDs; events addressed to unknown sessions are silently dropped (mitigates drive-by `POST`s from web pages to localhost).
- All investigation data (logs, hypothesis boards, field notes) lives in plain files under `~/.sherlog/` and never leaves your machine.
- The Case Board UI is read-only (GET only) and embeds all assets — it makes no external requests.
- Probes are one-line HTTP calls **you can grep for** (`2218/log/`); the cleanup gate exists so debug instrumentation never ships.

Anything that breaks one of these properties is a security bug.

## Reporting a vulnerability

Please **do not** open a public issue for security problems.

Use GitHub's private vulnerability reporting: **Security tab → Report a vulnerability** on this repository. Include reproduction steps and your sherlog version (`sherlog --version`).

You can expect an acknowledgment within a few days. Fixes are released as patch versions through the normal release pipeline (GitHub release + Homebrew tap).

## Supported versions

Only the latest released version receives security fixes. Update with:

```sh
brew upgrade sherlog
```

## Supply-chain notes

- Releases are built by GitHub Actions from signed tags via goreleaser; archives ship with a `checksums.txt`.
- The Homebrew formula in `neomodular/homebrew-tap` is generated by the release pipeline — formula changes outside a release are a red flag worth reporting.
